System OverviewFlexQL is a lightweight, high-performance, and persistent SQL-like database driver implemented entirely in C++17. It follows a robust client-server architecture:Server (flexql_server): A multithreaded TCP server utilizing asynchronous disk I/O for persistence and advanced memory structures for high-speed query execution.Client API (flexql_client): Exposes the required C-compatible APIs (flexql_open, flexql_close, flexql_exec, flexql_free) using an opaque FlexQL handle and communicates with the server via a custom wire protocol.REPL (flexql-client): An interactive command-line interface built on top of the client API.2. Storage Design (Memory & Disk Persistence)Designing the storage engine required balancing two conflicting constraints: the need for extreme throughput (to pass the 10M row benchmark quickly) and the need for durability and fault tolerance (data must survive server crashes).2.1 Disk Persistence (DiskStore & .bin Heap Files)A purely in-memory database violates the persistence requirement, and relying strictly on a Write-Ahead Log (WAL) leads to unacceptably slow recovery times via linear scanning.Design Choice: FlexQL uses a Hybrid In-Memory / Append-Only Disk Architecture.Schema Storage: CREATE TABLE statements write .schema binary files to disk.Data Storage: Every INSERT serializes the row data (length-prefixed columns and expiration timestamps) and appends it to a table-specific .bin heap file.The Trade-off (AsyncWriter): Writing to disk synchronously on every insert blocks the network thread and destroys throughput. To solve this, I implemented an AsyncWriter—a background C++ worker thread with a thread-safe queue. INSERT queries push serialized data to this queue and return immediately. The background thread flushes batches to the .bin files. This allows the server to achieve massive throughput (~540k - 890k rows/sec) while guaranteeing eventual durability.Fault Tolerance: On startup, load_from_disk() reads the .schema files to rebuild tables, then deserializes the .bin files to restore the exact state prior to a shutdown or crash.Note on 1TB Scalability: To maximize query speed for the 10M row benchmark, the system currently buffers the active dataset in memory. While this guarantees fault-tolerance, it is bottlenecked by physical RAM limits for datasets approaching 1TB. Future scale-out would involve migrating from full in-memory buffering to a Page-based Buffer Pool Manager with an eviction policy.2.2 Physical In-Memory Representation (Hybrid Row/Columnar)While string data is kept in standard Row-major format (std::vector<std::string> values) to optimize SELECT * projection, numeric data (INT, DECIMAL, DATETIME) is heavily optimized.Columnar Numeric Caches: Numeric columns are parsed at insert time and mirrored into parallel contiguous vectors (std::vector<double> numeric_column_values).Design Rationale: By keeping numbers in contiguous memory arrays, range queries (e.g., WHERE score > 90) avoid jumping through string pointers and parsing strings on the fly. This allows the CPU to leverage prefetching and evaluate conditions orders of magnitude faster.3. Indexing MethodFlexQL employs two distinct indexing strategies based on the Primary Key data type to guarantee $O(1)$ lookups.3.1 Integer Primary Keys: Custom RobinHoodIndexDesign Choice: A custom Robin Hood open-addressing hash map.Rationale & Trade-off: std::unordered_map uses a node-based, linked-list bucket structure. Under heavy load, this results in severe pointer-chasing and cache misses. My RobinHoodIndex stores all slots in a single flat std::vector. Robin Hood probing minimizes the variance in probe lengths (stealing from the "rich" to give to the "poor"), effectively eliminating worst-case $O(n)$ degradations. For INT primary keys, this yields $O(1)$ lookups with maximum CPU cache locality.3.2 String Primary KeysFor VARCHAR primary keys, std::unordered_map<std::string, std::size_t> is used as a reliable fallback, mapping the string key to the row's memory vector index.3.3 Numeric Range IndexingFor non-primary numeric columns, the engine utilizes the aforementioned parallel columnar arrays. WHERE queries with operators (=, <, <=, >, >=) execute a tight loop over these flat arrays, acting as an implicit clustered range index.4. Caching StrategyDesign Choice: LRU (Least Recently Used) Query Cache with MVCC-lite invalidation.4.1 Cache Key and CapacityThe raw SQL string is normalized (whitespace collapsed, keywords uppercased) to maximize cache hits. The cache holds up to 512 entries or a maximum memory footprint of 512MB. Queries exceeding 256MB are intentionally bypassed to prevent cache poisoning by massive full-table scans.4.2 Invalidation Strategy (Table Versions)Design Rationale: Traditional time-based caching serves stale data. FlexQL guarantees strict consistency. Every table maintains a monotonic version counter that increments on INSERT or DELETE. Cache entries store a snapshot of the table versions they referenced. During a lookup, if the stored versions do not perfectly match the current table versions, the cache entry is instantly rejected.Result: Repeated identical SELECT queries (e.g., dashboard aggregations) are served in microseconds without executing the scan, while still guaranteeing ACID-like read consistency.5. Handling of Expiration TimestampsEach inserted row can carry an expiration time via TTL <seconds> or EXPIRES <timestamp>.Design Choice: Lazy Evaluation (Tombstoning)The Problem: Deleting expired rows synchronously from a continuous .bin file or a std::vector is an $O(N)$ operation that blocks the database and fragments storage. A background Garbage Collection thread would introduce heavy lock contention.The Solution: FlexQL uses an expiry_flat vector parallel to the rows. During a SELECT or JOIN scan, the engine evaluates if (row.expires_at_unix != 0 && row.expires_at_unix <= now_unix()). If true, the row is skipped.Trade-off: This is an $O(1)$ check that completely removes the need for active background cleanup. Expired rows lazily consume disk/RAM until the server truncates/compacts the file, but execution speed is perfectly preserved.6. Multithreading & Concurrency DesignThe database server must support multiple concurrent client connections.6.1 Thread-per-Connection & Table-Level LocksInitial consideration: A global database lock (db_mutex_) was considered but rejected because it forces all clients into a single bottleneck. Client A inserting into users would block Client B reading from orders.Final Design: I implemented Table-Level Locking using std::shared_mutex inside each Table struct.Readers (SELECT): Acquire a shared_lock, allowing infinite concurrent reads on a table.Writers (INSERT/CREATE): Acquire a unique_lock only on the specific table they are modifying.6.2 Async Background WorkersDisk I/O is completely decoupled from the client request lifecycle. The AsyncWriter thread blocks on a condition variable until the transaction queue has data, allowing the main TCP threads to respond to the client immediately after writing to the in-memory data structures.7. SQL Execution & Advanced Trade-offs7.1 Hash Join vs. Nested Loop JoinFor INNER JOIN ... ON tableA.col = tableB.col, a naive Nested Loop Join operates in $O(N \times M)$ time, which would stall completely on 10M rows.Design Rationale: FlexQL dynamically implements a Hash Join. It identifies the smaller table and builds an ephemeral Hash Map of its join keys. It then streams the larger table against this hash map. This reduces the time complexity from $O(N \times M)$ to $O(N + M)$, yielding massive performance gains. Non-equi joins (<, >) fall back to optimized nested loops utilizing the fast columnar caches.7.2 Batch INSERT SupportNetwork round-trip times (RTT) are the biggest enemy of insertion throughput. The parser was optimized to handle batch inserts: INSERT INTO table VALUES (1, 'a'), (2, 'b'), (3, 'c'). Grouping inserts allows the AsyncWriter to flush data in 4MB disk blocks, significantly reducing kernel syscall overhead.8.
